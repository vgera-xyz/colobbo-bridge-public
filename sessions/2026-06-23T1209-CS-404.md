## Decisions (validated via /rthink → plan → operator approval)

1. **Stay supervised, don't flip** (operator pick). Build all four gap items; next session runs one supervised `run-loop.sh` cycle on the `~/dev` path before launchd is enabled.
2. **No native `/goal`** (claude-code-guide TIER-checked). CC has no built-in goal-loop primitive — the CS-404 ticket's stated premise is wrong. The launchd path is **`claude -p "<full procedure>"`** (Claude runs many tool turns inside one call until it raises the PR + STOPs). Hooks fire headless *as long as `--bare` is never passed*; that preserves the kill-switch/commit-on-stop control model unattended.
3. **launchd plist is an inert in-repo template, not an install** — Hard Rule #1 bars writing user-global (`~/Library/LaunchAgents/`) from this tree. Operator installs at the flip.
4. **Permission posture** — tight `permissions.allow` allowlist + `acceptEdits`, NOT `--dangerously-skip-permissions`; kill-switch is the hard backstop. The exact allowlist is the #1 thing the supervised cycle tunes.
5. **A 4th flip-prereq surfaced** — headless CC auth (`ANTHROPIC_API_KEY` or `claude setup-token` → `CLAUDE_CODE_OAUTH_TOKEN`); no interactive OAuth exists unattended. Documented, not provisioned.

## Lesson captured

The CS-404 ticket bakes in "native `/goal` driver" as the architecture's core piece — but verifying the harness reality (claude-code-guide) showed no such primitive exists. The honest substrate is `claude -p` + the per-worktree hook control model (which *does* work headless). Surfacing the premise gap before building avoided shipping a wrapper around a non-existent feature. Same shape as the migration sessions' "chat-side premise lost to TIER-1" corrections — falsify the premise first.
