### Pattern 3 — Tag-before-pull (orphan-after-rebase, self-corrected)

Created `v2.12.0` tag at local HEAD before pulling. `git pull --rebase` then brought in a graphify auto-refresh commit (`d9fc968`) from origin/main. After rebase, my v2.12.0 tag still pointed at the pre-rebase orphan commit (`ddb80ac`, same tree as new HEAD `61a095d` but with stale parent chain). Origin received the orphan tag via `git push origin --tags` before the rebase.

**Resolution:** Self-corrected by CC. Surfaced two options to Vipin (delete-and-retag OR leave orphan); Vipin chose delete-and-retag. Executed: `git push origin :refs/tags/v2.12.0` (delete remote tag) → `git tag -d v2.12.0` (local) → `git tag v2.12.0 HEAD` (re-tag at correct commit) → `git push origin v2.12.0` (push corrected). No `--force` flag needed; clean recovery.

**Memory updated:** `feedback_always_pull_latest.md` extended with pre-tag-pull clarifier ("Pull also BEFORE TAGGING, not just before EDITING. The auto-refresh workflows ... silently push commits to colobbo-agent-system main between manual sessions; assume origin has moved on every session boundary.").
