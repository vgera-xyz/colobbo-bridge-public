### Supervised cycle 1 → caught a real bug
The loop returned a confused STOP (no ingest). Root cause, verified against the filesystem: **`run-loop.sh` never `cd`-ed into `$WORKTREE` before `claude -p`**. Claude Code discovers `.claude/settings.json` from the process CWD, so the nested agent inherited the caller's CWD (the `cs-404-flip` agent-system worktree, not the api_node worktree) and loaded the **wrong** project settings — meaning the wired permission allowlist AND the control hooks (kill-switch / steer / commit-on-stop) were **all inert**. The ingest (`curl` / Sentry-MCP) was permission-gated in headless mode → the agent correctly refused to invent work and STOPped. It **failed safe only by luck** (the same broken CWD that killed the allowlist also blocked all actions). No PR, no code change.

### The fix
One-line subshell: `( cd "$WORKTREE" && claude -p "$PROMPT" --permission-mode acceptEdits )`. Post-raise steps use `git -C "$WORKTREE"` (absolute) so the `cd` is self-contained. `bash -n` clean. Commit **`3f44ab8fb0`**, pushed to `cs-404-flip` (updates PR #36) with a body documenting the gate finding.

### Supervised cycle 2 → clean
With the fix, the allowlist loaded, the REST ingest succeeded, and the loop adjudicated the live queue and returned a clean **NO_FIX** (no branch, no PR — correct). This proves the headless `claude -p` orchestration end-to-end for the no-qualifying-issue branch.
