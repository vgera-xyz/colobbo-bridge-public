### Phase 2 — build + verify

Wrote `scripts/shape-sp-dump.mjs` (418 LOC, stdlib-only Node, header-heavy docblock per `feedback-pragmatic-agent-over-substrate-evolution` ~3x baseline budget). Two real bugs surfaced + fixed during integration:

1. **structural_closure_failed on SP_CustomerContactDetailsGet** — over-strict "last meaningful line must be END/GO" check tripped on inter-block SQL Server scripting boilerplate (`/****** Object: ... ******/` + `SET ANSI_NULLS ON` + `GO`) that the slicer includes in the previous block's body. Fix: relax to "balanced BEGIN/END (load-bearing) + has at least one END line"; drop the strict last-line-terminator check. The truncation guarantee preserved by the balance check (a marker-inside-truncated-body would leave unbalanced BEGIN/END).
2. **structural_closure_failed on SP_CustomFieldDefBulkReorder** — `BEGIN TRAN` was counted as a BEGIN with no matching END (it pairs with COMMIT/ROLLBACK, not END). Fix: exclude `BEGIN TRAN|TRANSACTION|DISTRIBUTED` from BEGIN count via negative lookahead. `BEGIN TRY` + `BEGIN CATCH` still counted (they pair with END TRY + END CATCH).
3. **exact_match_gate mismatch (609 vs 991 expected)** — over-greedy literal-stripping. Apostrophe inside `--` comment ("there's") opened a literal-match the regex closed many lines later, eating CREATE PROCEDURE statements + BEGIN/END words in between. Fix: STRIP COMMENTS FIRST (block + line), THEN strip literals. Order matters.
4. **lenient regex needed** — strict line-anchored `^CREATE\s+(?:PROCEDURE|PROC)\b` missed 9 SPs with leading whitespace before CREATE (system `sp_*diagram` + `SP_SORSubGroupGet_ByGroupIds` + `SP_TrackerSiteUpdateV3`). Switched to `^\s*CREATE\s+...` lenient per-block detection. All 1000 markers map 1:1 to real procs.
