# CS-147 — pre-PR CI-conformance gate (the #278-landed-red fix), proven end-to-end

## Summary

Cycle-1's PR #278 was `evaluator-PASS` (the `ilike`→exact-match fix genuinely worked vs real data) but landed **red on CI**: the required `lint-changed` gate failed because the loop **never ran CI locally before raising**. This session added a **pre-PR CI-conformance gate** to the sentry-fix **builder** (the pr-raiser — there is no separate `pr-raiser` file; the builder opens the PR). The gate reads the integration branch's **required** checks live from branch protection (never hardcoded), replays them locally, and only raises once green — `UNVERIFIED`-if-a-required-check-can't-run-locally, **never-raise-red**, bounded fix-loop (max 2 rounds) then escalate. The **evaluator subagent is unchanged** (frozen — proven discriminating in cycle 1; verified byte-identical). Then **proved it**: re-ran the corrected loop on #278's branch via the loop's own builder machinery (not a hand-patch) → the gate caught `lint-changed` red, the builder cleared the 4 `no-unused-vars` errors minimally + behaviour-preservingly, and **both required checks went green**. PR #278 is now fully green, mergeable, still `evaluator-PASS`, OPEN (awaiting @jthakkar1981 — the loop never auto-merges, Rule #1).

## What was done

### The gate (agent-system, commit `9265a79890`, on `main`)
- **New `loop-harness/sentry-fix/ci-conformance.sh`** — read-only discovery helper. Reads required check contexts from branch protection (classic `required_status_checks` → repository-ruleset fallback) and resolves each context to the workflow file that defines it. `bash-jq-reflexes`-compliant (`jq -e` presence checks, no `|| true` on substitutions feeding correctness checks). shellcheck-clean. Verified against api_node `development` → prints exactly `typecheck` + `lint-changed`, `SOURCE=classic`. **Never hardcodes which checks gate a repo** — the whole point (CS-401 added these two checks; other repos differ; a hardcoded list silently goes stale).
- **`builder-prompt.md`** — new "Pre-PR CI conformance" stage between the evaluator-verdict/label step and the Terminal PR-raise: discover required checks dynamically → replay each reproducible check locally on the PR branch (compute changed-file sets exactly as the workflow does) → classify reproducible-green / reproducible-RED / not-locally-reproducible → decide (all green → raise with surface label; any red → bounded fix-loop, never raise red; any not-runnable → raise `UNVERIFIED — human test required` naming it). **Anti-gaming hard rule:** make the *code* conform; never edit the gate (eslint config / tsconfig / workflow YAML / branch protection) to pass a check.
- **`routing.md`** — documents the CI-conformance label dimension and how it composes with the functional (evaluator) axis on the single PR label. Backend's "no `UNVERIFIED` for an unreachable DB" rule (functional axis) is unchanged; the CI axis is a separate legitimate `UNVERIFIED` reason.
- **`README.md` + `RUNBOOK.md`** — flow line + Files list + run-procedure step + a note recording the #278 gap and this proof.
- **`evaluator.md` — untouched** (`git diff` empty). The gate lives in the builder only.

### The proof (api_node PR #278, commit `6ad16a6` on `cs147-sentry-fix-1`)
- Dispatched the builder machinery on the existing `cs147-fix` worktree to run **only** the new gate on the already-evaluator-PASS fix.
- Gate found `typecheck` green, `lint-changed` **red** — 4 `@typescript-eslint/no-unused-vars` in `api-apps.service.ts` (unused `NewApiKey` import + three `_kh` rest-destructure omissions; the gate lints the *whole changed file*, so the `ilike` fix inherited that pre-existing key-function debt).
- Builder fix: removed the unused import; replaced the three `_kh` throwaways with a typed `stripKeyHash(row): Omit<ApiKey,'keyHash'>` helper — `keyHash` still excluded from every returned object, all queries/side-effects unchanged, no gate/config edits, one file. Re-ran both checks → **green**.
- **Branch-divergence handled correctly:** the PR branch had advanced (someone merged `development` in → tip `103a136`, still red on `lint-changed`). The builder **refused to force-push or autonomously rebase** and reported. Verified the divergence was a benign update-merge, **rebased** the single commit onto the new tip (clean), **re-ran the gate green**, **fast-forward pushed** (no `--force`).
- CI re-ran: `lint-changed` **FAILURE → SUCCESS**, `typecheck` SUCCESS. Full rollup green; `mergeable`; label `evaluator-PASS`; OPEN.
