# CS-373 M8 increment-1 — cross-Lobe contract gate dual-mode (tag v2.48.2)

## Summary

Shipped **M8 increment-1** — the enabling substrate for the CS-373 consumer-side tolerant-reader migration (each consumer's `consumes_<X>_schema` semver-range pin → `consumed_fields.<X>: [...]` field-set declaration). M8 is a **multi-session, lowest-coupling-first track** (derivation §7: Frontend → Mobile → Architect → Backend → Database → Test → Docs → Security → Code Reviewer LAST; ~5 sessions). This increment ships **only the dual-mode gate** so the per-Lobe migration can proceed Lobe-by-Lobe without breaking CI. **Zero Lobes migrated** this increment — every pin stays semver-range, the gate behaves exactly as under M7 on the real run; the new branch is synthetic-proven. M8 implements **CS-359 Fix-1** (tolerant-reader) and absorbs **CS-352** (Pattern-B declarative-consumption) per the derivation. PATCH ship (gate-only).

## What was done — `scripts/verify-lobe-compat.sh` dual-mode

Each REGISTRY entry is validated in **whichever form the glaze declares**:
- **OLD form** `consumes_<X>_schema` (semver-range) → invariants (α)(β)(γ) unchanged.
- **NEW form** `consumed_fields.<X>: [...]` → invariant **(δ)**: producer schema exists; the field-list agrees across both glazes; is a non-empty array; and **every listed field's top-level segment exists in the producer's output-schema `.properties`** — the REMOVE/RENAME detector. A producer dropping/renaming a consumed field FAILs as **drift (class 3)**; additive producer changes pass silently (= tolerant-reader). The consumed_fields path is **derived** from the registry's semver path (`consumes_<key>_schema` → `consumed_fields.<key>`), so migrating a Lobe needs **no registry edit** — only the glaze block flips form.
- **BOTH** forms present for one relationship → error (half-finished migration); **NEITHER** → error (pin vanished); both class 1.
- The **completeness forcing-function** is now dual-mode: it enumerates both `consumes_*_schema` and `consumed_fields.*`, normalises the latter to canonical form, and confirms every relationship (either form) is registered-or-excluded.

## The key discovery (validated via /rthink — the active-plan one-liner was falsified)

The active-plan framed M8 per-Lobe as "swap glaze semver-range → consumed_fields[]". **Reality:** each of the 9 consumer LOBE.md files references its pin via `{{glaze.<lobe>.compatibility.consumes_<X>_schema}}` **compose-time tokens** AND performs a **runtime `schema_version` version-gate** in its methodology ("read upstream's `schema_version`, check against the range, if outside → `incompatible_<X>_schema` error → stop"). True consumer-side tolerant-reader (derivation §11 Test 2 — "**removes VERSION-coupling**, not ORDERING-coupling") **removes that runtime version-gate + the token-sites**, replacing them with `consumed_fields` + this build-time gate. So per-Lobe migration = glaze swap + LOBE.md version-gate removal + recompile + **`schema-incompat-*` fixture rework** (each consumer has a `schema-incompat-*` fixture whose expected outcome **flips error → tolerate**). That is why increment-1 is **gate-only** (orthogonal to the consumer-behaviour change, additive, zero-RED, ≤1 session) and the behaviour change is deferred to the per-Lobe increments.

## OPERATOR DECISION required before increment-2 — Fork B

Increment-2 = migrate **Frontend** (lowest-coupling, 1 pin: `consumed_fields.arch = [components, file_mapping, interface_contracts]`, all in architect `required[]` → no gate RED). Before it, decide **how** the consumer-side tolerant-reader behaviour is implemented across the 9 consumers:

- **(a) RECOMMENDED — a new Layer-0 `tolerant-reader@1.0.0` shared fragment** that all 9 consumers inherit via `includes:` (the derivation's §2.F hypothesis; the `lobes/_shared/` mechanism + the emit-discipline/ADR-032 precedent make it viable). Uniform tolerant-reader semantics; the `schema-incompat-*` fixture rework becomes uniform. Cost: build the fragment + wire 9 `includes:` + recompile all 9.
- **(b) per-Lobe bespoke LOBE.md rewrites** — 9× individual version-gate removals. Error-prone, inconsistent, no single source.

This fork determines the shape of the remaining ~4 M8 sessions.

## Lesson captured

The active-plan one-liner hid a fleet-wide consumer-**behaviour** change behind what read as a metadata swap. Grepping the actual LOBE.md token-sites + runtime version-gate (not trusting the summary) is what surfaced it — and turned M8 from "edit 23 glaze lines" into "remove the runtime version-gate from 9 consumers + rework their schema-incompat fixtures, via a Layer-0 fragment." Front-loading the orthogonal, zero-risk gate enabler (increment-1) isolates that behaviour change into deliberate per-Lobe increments with an explicit operator fork — rather than smuggling it in under a glaze edit.
