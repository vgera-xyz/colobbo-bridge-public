## Routing observation (informational, not a gate)

The smoke's a-priori expectation (per the task brief) was `web-angular` +
`api_node`. The Lobe instead routed to **`webDotnet` + `webAngular`** and did
**NOT** select `apiNode`. This is the Lobe being *more correct than the
expectation*, not a defect:

- It read the actual code-intel trace and found the Scheduler backend is the
  legacy .NET `Colobbo-Web` (`SchedulerController.cs` / `ScheduleService.cs`,
  SP-backed: `SP_SchedulerSettingGet` / `SP_ScheduleSettingInsertUpdate` /
  `SP_BookingScheduleInsertUpdate`), with the settings/Create-Schedule UI in
  `web-angular`.
- This matches [[../../migration/progress|migration progress]]: Scheduler /
  Dispatch is **Tier-1, ~95% .NET** — it has not been ported to `api_node`.
  Routing to `apiNode` would have been the *wrong* answer.
- Key→name: `webAngular → web-angular`, `webDotnet → Colobbo-Web`.
- Appropriate risk flags fired: `tier1_module` (high),
  `high_risk_owner_module` (high), `sp_refactor`, `sp_migration_required`.

Net: check #4 passes on its actual intent (real glaze-derived identifiers, no
placeholder tokens); the deviation from the literal "api_node" expectation is
the Lobe correctly grounding on the trace.

## 3 findings for Friday substrate batch

1. **Glaze keys (camelCase) vs names (snake/kebab) — doc gap.** A correct
   PlanResult uses glaze.repos *keys* (`webAngular`, `apiNode`, `webDotnet`),
   not the `.name` values (`web-angular`, `api_node`, `Colobbo-Web`). This
   tripped the smoke's check-#4 wording and is undocumented at the contract
   surface. Action: document the key-vs-name convention in the Planner LOBE
   §Outputs and/or output-schema descriptions so consumers/verifiers don't
   expect `.name` form.
2. **Intent classifier matched `bugfix` on a negating-context phrase.** The
   rule-based classifier scored `bugfix` because the description says *"not
   part of the current **bug fix** batch"* — a negation. The ticket is
   explicitly an enhancement. The Lobe self-flagged this in `feature_summary`
   (good failure-transparency) but routed the `bugfix` pipeline. Action:
   keyword classifier needs negation/context awareness, or an
   enhancement/feature keyword cluster, before more tickets are run through it.
3. **Planner version drift.** Emitted `planner_version: "1.0.1"` while
   `lobes/planner-lobe/LOBE.md` frontmatter is `version: 1.0.2`. Semver
   *pattern* gate passes; the *value* is stale vs the LOBE source. Action:
   reconcile so PlanResults self-report the running version.
