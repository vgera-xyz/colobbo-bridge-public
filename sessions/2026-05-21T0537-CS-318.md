# CS-318 — get-jwt.sh Cognito wrapper + CS-322 sizing decision

## Summary

Short tooling session: built `scripts/get-jwt.sh`, an AWS Cognito access-token primitive for the dedicated `cs-agent-test` identity, for CS-318 parity testing against api_node. Ships pending-seed (no credential yet — awaiting Jignesh). Forward-compat with CS-322 headless GHA via a `--mode=admin` switch sharing the same `AuthenticationResult.AccessToken` parser. Also produced the **CS-322 sizing decision note** (active-plan §CS-322 sizing — see Step 1a) reframing yesterday's 200-300M token cumulative envelope as a conservative ceiling, with mature-substrate midpoint at ~80-120M.

## What shipped

**1 commit** on `main`: `efd072103 feat(auth): add get-jwt.sh for Cognito refresh-token + admin flow [CS-318]`. Pushed to origin.

`scripts/get-jwt.sh` (336 lines, mode 100755):

- **Two modes**, single shared response parser:
  - `--mode=refresh-token` (default, laptop): `REFRESH_TOKEN_AUTH` via `aws cognito-idp initiate-auth`; refresh token from macOS Keychain.
  - `--mode=admin` (CS-322 GHA path): `ADMIN_USER_PASSWORD_AUTH` via `aws cognito-idp admin-initiate-auth`; username + password from env (GHA secrets) + AWS IAM creds.
- **Stdout contract**: access token only, NO newline. Designed for `JWT=$(./scripts/get-jwt.sh)` capture without trailing whitespace.
- **Errors stderr-only**: tokens never logged, even in `--print-exp` debug mode (which emits only `exp` + `ttl`).
- **Exit codes documented** in header block: 0 success | 1 usage | 2 Keychain missing (re-seed) | 3 Cognito API error | 4 refresh token expired | 5 aws CLI missing | 6 cache parse error.
- **SECRET_HASH path** is a conditional branch gated by `COGNITO_APP_CLIENT_HAS_SECRET=true` — defaults to OFF per the request to Jignesh for a no-secret app client; dead code if has-secret stays false.
- **Cache** at `~/Library/Caches/colobbo-cs-agent/jwt-cache.json` (chmod 600, 5-min skew margin); **lock** via `mkdir`-as-mutex (macOS-compatible, no flock dep); double-check-after-lock to avoid serialising the cached fast path.
- **Config env-file** at `~/.colobbo/cs-agent.env` (chmod 600, out-of-repo by design — never beside the script). Created automatically by `--seed` if missing.
- **No secrets ever on disk outside Keychain**: refresh token, client secret, user sub all go via `security add-generic-password`. Gitleaks/TruffleHog clean by construction.
- **bash-jq-reflexes compliance**: `jq -e` for presence checks, `local var; var=$(cmd) || fail ...` pattern (avoids set -e gotcha on local-with-assignment), temp-file + mv for cache writes (never `>` redirect to source).

## Pending-seed state

Script is **content-complete but credential-pending**. Every invocation will exit 2 with re-seed instructions until `./scripts/get-jwt.sh --seed` runs with Jignesh's refresh token. That is the correct initial state, not a defect.

When Jignesh responds with credentials, the seed flow:

1. Fill `~/.colobbo/cs-agent.env` with `AWS_REGION`, `COGNITO_USER_POOL_ID`, `COGNITO_APP_CLIENT_ID`, `COGNITO_APP_CLIENT_HAS_SECRET=true|false`.
2. `./scripts/get-jwt.sh --seed` → paste refresh token at the hidden prompt.
3. If `has-secret=true`: also prompt for client secret + user sub (decoded from a prior access token).
4. Script auto-verifies by re-exec'ing with `--no-cache --print-exp`; success path emits `exp=<unix-sec> ttl=3600s (fresh)` to stderr + the token to stdout.

## Questions sent to Jignesh (operator-relayed alongside the cred request)

1. Does the `cs-agent-test` Cognito app client have a client secret? (No → SECRET_HASH branch stays dead code. Yes → §6 branch activates.) **Requested: no-secret app client.**
2. Refresh token validity period configured on the app client? (Default 30d; some pools set 1d.)
3. MFA enforced on the test user? (Should be NO for a programmatic identity.)
4. Test user role + `selectedCompanyId` in dev env — at minimum scoped to the same role as the legacy .NET caller for CS-318 parity testing.

## CS-322 sizing decision (the second substantive output)

Yesterday's 200-300M token cumulative bound for COL-205 (660 endpoints) was conservative-by-construction. WorkOrderType GetAll consumed ~300-400k tokens per endpoint across 3 review rounds — but 3 of those were **substrate-class fixes** (Round-1 pre-redesign defect, Round-2 recursive SP-name failure, Round-3 dependent-SP body-read), not endpoint complexity. Mature-substrate typical = 1 round; occasionally 2 for the first endpoint in a new module shape.

**Distribution sketch (mature substrate):**

| Class | Per-endpoint | Share of 660 |
|---|---|---|
| Composition-heavy KEEP (WorkOrderType-class, 1 round) | ~150-200k | ~10-15% |
| Single-SP CRUD KEEP | ~50-80k | ~60-70% |
| Simple MOVE-mode | ~30-50k | ~15-20% |

**Cumulative bands for 660 endpoints:**
- Optimistic floor: ~80M tokens
- Midpoint: ~100-120M tokens
- Conservative ceiling (with ~10% Round-2 substrate-edge-case bumps): ~150-180M tokens
- Pessimistic upper: ~200M tokens (still well under the original 200-300M)

**Run-pattern mix recommended:**
- **Pattern A (Max OAuth headless via GHA, CS-322 baseline)** for the predictable ~5/day baseline ($0 incremental; rate-limited by 5h-window).
- **Pattern D (API-key bursts)** when batch-1 (5 modules parallel per COL-205) needs to clear faster than Pattern A's 5h-window allows. ~$1-2k total at midpoint — trivially affordable.
- **Pattern E (devs-on-own-CC)** for parity-test reviews + ad-hoc validation. Combined ~8-12/day hits the 8.3/day average for 2026-06-30 deadline.

**Verdict: tolerable.** Binding constraint is wall-clock + Jignesh reviewer capacity, NOT tokens. Max-plan 5h-window cap is the only number to verify before locking CS-322 sizing (I don't have an exact figure in this session's context — flag for verification before CS-322 build kickoff).

This decision lands in `wiki/strategy/active-plan.md` via Step 1a below (per ADR-034 §1.D2 ownership boundary — Step 1a is the canonical mutation point).
