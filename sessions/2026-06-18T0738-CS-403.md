## Summary

Long continuous colobbo-devstack session. Dominant arc: diagnosed and fixed the post-login dashboard "Network error" on the local Docker stack, then recovered the stack after a full rebuild + re-seed. The decisive technique was **driving the real browser** (Claude-in-Chrome) instead of inferring from server logs — that revealed the failure was in **login's .NET call**, not the dashboard. Root cause: web-angular's `apiUrl` was missing the `/api/` base path, so `MobileAPI/UserLoginV2` 404'd and login never completed. Verified the correct base against the *working* deployed app (`dev.colobbo.com` calls `…/api/MobileAPI/*` → 200, CORS `*`). A cascade of SSDT seed-drift then surfaced and was patched via the CS-403 band-aid sidecar. All fixes are devstack-local; no product-repo code changed except the already-merged metabase removal.

This session is **not** a CS-403 completion — it extended the CS-403 drift band-aid (added `tblRole`) but the SSDT reconcile CS-403 tracks remains open. CS-403 left in its pre-session state (no auto-transition).

### Dashboard login — root cause + fix (devstack-local)

- **Login's `.NET` `MobileAPI/UserLoginV2` call had a malformed URL.** `apiUrl` was the bare host (no trailing slash, then later `/` without `/api/`), and bare endpoints concatenate onto it. The correct base is `https://demo-webapi.colobbo.com/api/` — confirmed by reading the working deployed app's network calls (`…/api/MobileAPI/GetUserCompanyDetailsV1/1` → 200) and by curl (`/api/MobileAPI/UserLoginV2` → 200, CORS `access-control-allow-origin: *`, localhost allowed). The `ENVIRONMENT-SETUP.md` value (`…/` without `/api/`) is stale.
- Fixes: `309282b` (node + milestone base URLs → `/api/`), `6a1dd56` (trailing-slash normalize in `gen-web-env.sh`), `80fecb3` (`WEB_DOTNET_API_URL=…/api/` + template guidance).
- **Cascading SSDT seed-drift (CS-403 band-aid grew):** `cognitoUserId` +5 `tblUser` cols (`1beca72`), `SP_UserGet` 3-param proc (`67d9ca5`), `tblRole` `roleType` +5 cols (`80fecb3`). After these, all `/api/dashboard/*` calls return 200.
- **Remaining zeros are not a bug** — the user resolves to `companyId=1` via the remote `.NET`, but the synthetic seed data lives under tenant `9001`, so the queries correctly return 0 rows. This is the seed-data gap (devstack issue #1).

### Recovery after a full rebuild + re-seed

- Re-applied the live-only `tblUser` id=1 sub-mapping (`cognitoUserId 49ded4c8-…`) + `tblUserPreferences` (company 9001).
- The re-seed came up **12 tables short** (non-deterministic FK multi-pass loader; `tblUserPreferences` among the missing). Recovered all 12 → full **337/337**.
- web-angular wouldn't compile: `git status` itself threw `error: short read while indexing` on ~12 files — the **macOS/iCloud file-corruption gremlin** (truncated/evicted placeholders, not deletions). `git checkout HEAD -- :/` restored them from the intact object store.
- After restore, the build cascaded 69× `TS-996002 'SharedModule' does not appear to be an NgModule` — the **primary** error was a single `-35` (FUSE sass-read EAGAIN, container up 39h). A **web-angular restart** (fresh mount handle) cleared it → `Application bundle generation complete [43.3s]`, 0 errors. HEAD (`e5b5eb6`) is the same commit that compiled clean earlier — code was never the problem.

## Durable lessons

- **Browser-driven diagnosis beats server-log inference for client-side failures.** Server logs showed dashboard-path errors that only mattered *after* login; the real blocker (login's `.NET` URL) was only visible by driving the actual app and reading network + console.
- **Verify config against the working system, not stale docs.** The `/api/` base came from the live deployed app's network calls, not `ENVIRONMENT-SETUP.md`.
- **The iCloud/macOS gremlin recurs** (file truncation + Docker FUSE `-35`). Cures: keep the repo folder "Downloaded"/pinned out of iCloud "Optimize Storage"; a periodic `docker compose restart web-angular` clears a degraded-mount `-35`.
- **The MCP browser context is isolated** from the user's normal tabs (separate cookie/storage); a Cognito SSO "continue" works only if that context already holds the session. Password entry is never done by the agent.
