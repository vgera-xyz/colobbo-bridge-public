## Correction 2026-05-29T0658 ‚Äî CS-390 re-architected to the corrected operator+Jignesh model

The CS-390 ship recorded above (`904040fef`, gate-only curated-schema + `SCHEMA-PROMOTION-NEEDED`) used the **wrong model**. Operator aligned with **Jignesh** (api_node convention owner) on the corrected model; re-architected in commit `0bd1ccfc2` (forward correction ‚Äî CS-390 stays Done; the original audit trail above is preserved, the forward-state is corrected here).

**Corrected model (plain terms):**
- **Jignesh owns all Drizzle schemas.** He maintains `mssql-schema.ts` with full info (FK, identity, nullability, datetime2 precision) and will add a PostgreSQL equivalent (`postgres/schema.ts`, pending). The worker READS them only ‚Äî NEVER creates, modifies, or auto-promotes schema files. This retires the entire gate-only / curated-vs-dump / `SCHEMA-PROMOTION-NEEDED` apparatus from the prior ship.
- **MOVE** = a native Drizzle query against the Jignesh-owned dialect schema (MSSQL source = `mssql-schema.ts`, not the curated subset). Emits NO schema.
- **KEEP** (rewritten) = a Drizzle service calling the existing SP via `db.execute(sql\`‚Ä¶\`)` ‚Äî PG `SELECT * FROM <fn>(...)`, MSSQL `EXEC <SP> @p=...`; no schema import. Fallback to `executeStoredProcedure` (49 sites) if the beta's recordset return proves unreliable (worker confirms at Step 4a Context7).
- **Schema-gap split (D3):** MOVE + missing schema/table/column ‚Üí **STOP + `SCHEMA-MISSING`** (never silently degrade to KEEP; operator messages Jignesh, re-runs after). KEEP needs no schema ‚Üí proceeds; Step 9 notes incidental gaps for batch follow-up.
- **Parallel-endpoint lifecycle (Boundary 19):** the worker emits a NEW route + controller method + service ALONGSIDE the legacy endpoint; never removes/modifies the legacy route handler, legacy SP, or any legacy code. Web integrates the new endpoint (~1 month) ‚Üí mobile (~1 month) ‚Üí legacy deprecated ‚Äî operator-managed; the worker does not gate on it.

**Smoke-blocker dissolved:** the "BootsOnGround inaugural smoke is BLOCKED" section above no longer holds ‚Äî under the corrected model the worker reads `mssql-schema.ts` (where all 6 BootsOnGround tables ARE present). The smoke is now gated only on Jignesh keeping `mssql-schema.ts` precise, not on a curated-schema gap; the "curation strategy + 4-layer gap inventory" open-work item is dissolved.

**Verification:** Context7-confirmed PG `db.execute(sql\`‚Ä¶\`)` (returns rows; injection-safe params); MSSQL `execute`+`sql` surface confirmed (api_node `tx.execute` precedent) with EXEC-recordset return-shape deferred to the worker's Step-4a check.
