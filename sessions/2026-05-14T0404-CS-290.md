# CS-290 — Ensemble PR review judge ship

## Summary

Shipped the CS-290 ensemble PR review system end-to-end across one canonical repo + three product-repo propagation PRs over a ~12-hour window spanning 2026-05-13 evening → 2026-05-14 early morning. The original ticket scope (3-model parallel ensemble to mitigate Petri judge-family-bias) was expanded to bundle three adjacent PR-review workflow features: synchronize trigger, path filter (Feature 2), and historical context across re-reviews (Feature 3). All four features shipped, smoke-tested live on web-angular's `development-stage` branch, and validated functionally — including aggregation via Anthropic's structured-outputs beta producing strict-JSON `{agreements, disagreements, verdict}` output.

The ship was longer than estimated (initial plan: ~1.5–2 hours one-shot) because three real bugs surfaced through downstream validation — none caught by `/rthink` upstream. Each bug forced a small hotfix cycle: canonical patch → push → product-repo hotfix PR → reviewer merge → re-smoke. The "layered defense" pattern (smoke test, ensemble cross-family agreement, Sentry independent review) is the lesson — codified at session close as **Pattern T (API/CLI contract-shape assumption errors)** in the methodology patterns catalogue.

## Pattern T — API/CLI contract-shape assumption errors (codified)

Added to `wiki/raku-meta/state-drift-patterns.md`. Two sub-cases:

- **T.1 — API response field-shape mismatch.** Different endpoints for the same logical concept return different field names. `gh pr view --json comments` (GraphQL) returns `author.login = "github-actions"`; `gh api repos/.../issues/.../comments` (REST) returns `user.login = "github-actions[bot]"`. Filters built from memory of one endpoint silently fail against the other.
- **T.2 — CLI runtime-context dependency.** Repo-aware CLIs (`gh`, `git`) need either `actions/checkout` OR explicit `--repo` flag. Without one, the CLI fails at runtime with cryptic git-context errors.

**Layered-defense meta-lesson** (the most important takeaway): all 3 gaps were caught downstream of `/rthink`, not by it:

1. Ensemble cross-family agreement caught its own contradiction on Feature 3 smoke
2. Sentry independently caught Fix #3 (PR title injection) — Pattern A (two-tool convergence) instance
3. Smoke testing caught both T.1 and T.2 misses — empirical beats static for contract-shape gaps

The strategic-level fix is investing in multiple validation surfaces, not perfecting any single discipline. `/rthink` SKILL.md strengthened at session close:

- Point (d) Check: `Grep/Glob existence AND probe actual behavior; REST vs GraphQL field names differ; existence ≠ behavior (Pattern T)`
- New point (j): `Does any CI/CD step invoke a repo-aware CLI (gh, git)? Verify each call has actions/checkout step OR --repo OWNER/NAME flag (Pattern T.2)`

Checklist count bumped 9 → 10 in three places (header, summary template, close-discipline statement).

**The class.** Two sessions in parallel on the same shared resource (Pattern-letter sequence in `state-drift-patterns.md`) without explicit coordination → collision risk. Pattern G's "verify filesystem state" reflex works at the moment of grep, but a parallel session can ship between the grep and the commit, invalidating the verification. Tonight's mitigation was a second re-verification at commit-time. A future Pattern letter (or extension of an existing one) could codify the discipline more durably — explicit lock, letter-assignment registry in glaze, or session-id tag on draft entries. **Filed as observation in this bridge entry only; no Linear ticket spawn per NO TICKET SPAWN principle.** When a third instance fires, that's the cue to formalize.
