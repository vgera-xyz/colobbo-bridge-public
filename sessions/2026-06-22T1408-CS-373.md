# CS-373 M6 ÔÇö ADR-012 capability-profile migration for the 10 pre-ADR-012 Lobes (tag v2.48.0)

## Summary

Shipped **M6** of the CS-373 Phase-1 sequence: the 10 previously-pre-ADR-012 Lobes now declare an **ADR-012 Layer-1 capability profile** in their `LOBE.md` `capabilities:` frontmatter, matching Backend's already-shipped state (CS-340). **Post-M6, all 11 Lobes carry a capability profile.** Per CS-373 ┬º10.8 this is the deliberately-lightweight increment ÔÇö a genome declaration ("mechanically similar to adding the Layer-0 `includes:` entry"), NOT the heavier per-Lobe full-adoption (building audit operations + output-schema bumps + fixtures) that the CS-340 Backend POC was. The derivation explicitly scoped M6 down from orchestration-architecture's "~1-2 sessions per Lobe" to "~0.3-0.5 session per Lobe" precisely by drawing this line.

Per-Lobe profiles (atoms from the **closed ADR-012 ┬º3 enum**, each grounded by reading the LOBE.md body ÔÇö what the Lobe *actually implements*, per ADR-012 ┬º4 "declare its full implemented profile"):

| Profile | Lobes |
|---|---|
| `[read_code, produce_plan]` | planner, architect, frontend, database, mobile, memory, test-writer, doc-writer |
| `[read_code, produce_findings]` | security, code-reviewer |
| `[read_code, produce_plan, produce_findings]` (unchanged, CS-340) | backend |

The 8 `produce_plan` Lobes are spec/plan emitters ÔÇö they emit a structured Result (PlanResult / ArchResult / FrontendResult / DatabaseResult / MobileResult / SessionUpdateResult / TestResult / DocResult). The *actual* file/schema/test artifacts belong to the **out-of-scope L2 codegen sub-agents** (`write_new_files` / `produce_schema` / `generate_tests`), exactly as Backend omits `write_new_files`. Notably **Database ÔåÆ `produce_plan`, NOT `produce_schema`**: its LOBE.md states it emits a dialect-neutral spec and "does NOT specify dialect-specific DDL... those are the code-writer sub-agent's job." Security + Code Reviewer emit findings/verdicts ÔåÆ `produce_findings`.

## Decisions (validated via /rthink before building)

1. **Layer-1 only ÔÇö no glaze `enabled_capabilities` edit.** ADR-012 ┬º4 invariant sanctions absent `enabled_capabilities` ("no profile asserted, behaviour unchanged"). Layer-2 activation pairs with (a) a built multi-capability operation surface (none of the 10 has one ÔÇö they are single-op) and (b) the deterministic dispatch gate, deferred to **CS-324** (ADR-012 ┬º10). Reinforcing signal: Memory / Code-Reviewer / Doc have **no name-keyed glaze block** (`glaze.wiki` / `glaze.review` / `glaze.docs`), so Layer-2 isn't even a clean mechanical mirror. Backend has Layer-2 only because CS-340 built its `build|audit` operation requiring activation.
2. **No per-Lobe `version:` bump.** `capabilities:` is frontmatter (stripped before body compose) AND absent from compose's `frontmatterFields` list ÔÇö so compiled `agents/global/*.md` are **byte-identical** (Gate 14 unaffected) and the registry `composite_signature` tracks version+includes, not capabilities (Gate 19 unaffected). **Empirically confirmed**: `compose.sh deploy --all` + `registry-regen` produced zero diff beyond the 10 LOBE.md files. Backend's CS-340 MINOR bump was for the audit *operation* it built, not the declaration. Precedent for LOBE.md-edit-without-bump: `a4dbc9b67a`.
3. **`produce_plan`-as-spec-emit-atom ratified fleet-wide.** Backend's CS-340 left this as open note R1 ("if rejected the remedy is an ADR-012 amendment"). M6 applies it to 8 more Lobes; consistent with ADR-012 ┬º3 ("produce_plan = structured plan/**decomposition**") and *cleaner* for these Lobes than for Backend (they emit specs, never files). **No ADR amendment required.**

## What's next

## Lesson captured

The derivation deliberately distinguished **"adopt the capability profile" (Layer-1 genome declaration)** from **"full ADR-012 adoption" (operations + activation + gate)** ÔÇö and scoped M6 to the former. Reading the CS-340 commit message (not just the ADR) was what surfaced that Backend's `produce_findings` reflects a *built audit operation*, not the bare declaration ÔÇö which is why the 10 single-op Lobes correctly get a 2-atom profile, not a copy of Backend's 3-atom one. Grounding each atom in the LOBE.md *body* (not the description blurb) is what kept `produce_schema`/`generate_tests` off the spec-stage Lobes.
