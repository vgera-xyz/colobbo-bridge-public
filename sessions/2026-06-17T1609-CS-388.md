# CS-388 + CS-50 — two GitHub Actions cron issues diagnosed + fixed

## Summary

Diagnose-first session on two CI cron issues. **CS-388** (graphify-refresh): the Sunday cron *does* fan out to the 3 in-scope product repos correctly, but the `refresh` job had no job-level `if`-guard, so on schedule events it ran with an empty `TARGET_REPO`, failed at "Validate target repo", marked every Sunday run RED, and auto-filed a junk issue each week — fixed with a one-line guard. **CS-50** (weekly-security, 3 repos): web-angular was red ~3 months on an Angular peer-dependency **ERESOLVE**; api_node had **never run at all** since 2026-04-15 due to an **invalid top-level `concurrency` expression**; Colobbo-Mobile was green and only needed a deprecated-action bump. Three product-repo PRs opened (one per repo, correct integration branch + reviewer). The stale agent-system *source copy* of `weekly-security.yml` (diverged from all 3 live copies, synced to none) was retired. All diagnosis was evidence-quoted before any fix; the "ERESOLVE everywhere" premise held for **only** web-angular.

## What was done

### CS-388 — graphify-refresh cron (agent-system, pushed to main)
- **Verified the cron fires** for the **3 in-scope** repos (NOT 4 — Colobbo-Web is excluded by design, `.NET decommissioning`). Evidence: 2026-06-14 + 2026-06-07 each dispatched a green trio (`Colobbo-Mobile`/`web-angular`/`api_node`), confirmed via each run's `Target ok:` line.
- **Real bug:** the `refresh` job (`graphify-refresh.yml:38`) had no job-level `if`. On a `schedule` event (no `client_payload`/`inputs`) `TARGET_REPO` is empty → `case … *) exit 1` at "Validate target repo" → schedule run = FAILURE → the `if: failure()` "Open issue on failure" step fired weekly, filing junk issues **#33/#34/#35** ("Graphify refresh failed for "). The separate `weekly-trigger` job (the actual fan-out) succeeded throughout.
- **Fix:** added `if: github.event_name != 'schedule'` to the `refresh` job (commit `3ad79c7a2a`). The job now *skips* on cron; `weekly-trigger` is untouched. Closed #33/#34/#35 with a pointer to the fix.
- **Stale-premise finding:** CS-388's original premise ("`repository_dispatch` never fires") is no longer true — green `repository_dispatch` runs ~daily Jun 4–17. The auto-refresh-on-push chain works now; only the schedule-run wart remained.

### CS-50 — weekly-security across 3 product repos (3 PRs)
Investigation answer to "is this systemic false coverage?": **No.** web-angular fires (11 scheduled runs), Colobbo-Mobile fires (8), only **api_node = 0**.

- **web-angular → [PR #555](https://github.com/Colobbo/web-angular/pull/555)** (development-stage / @pjtech18). Red ~3mo at `npm ci --ignore-scripts` with a committed-lockfile **ERESOLVE**: 6 runtime `@angular/*` packages + `compiler-cli` stranded at `19.2.18` while `core`/`compiler` were `19.2.21`, and the stranded ones pin an **exact** `peer @angular/core`. Premise was "bump animations" but the census showed the *whole framework* stranded — animations-only would still ERESOLVE. **Operator chose to align the entire framework to `19.2.25`** (latest 19.2.x, caret-preserving = `ng update`); regenerated lock via `--package-lock-only`, strict reconcile clean, diff scoped to **exactly the 9 `@angular` packages** (no other deps moved). + `setup-node` v4→v6. Also affects `build-prod.yml` (plain `npm ci`) which would break on next release.
- **api_node → [PR #279](https://github.com/Colobbo/api_node/pull/279)** (development / @jthakkar1981). Lockfile **CLEAN** (`npm ci --dry-run`: 1065 pkgs, no ERESOLVE). **Real bug = the scan never ran:** top-level `concurrency.group: weekly-security-${{ matrix.app }}` references `matrix` at the **workflow level where it is not in scope** → invalid workflow → GitHub never registered the schedule (0 schedule + 0 dispatch runs since 2026-04-15; the 50 visible runs are `push` events from a contaminated feature branch; the workflow even registers `name` as the file *path*). **Fix:** moved `concurrency` into the `security-scan` job (matrix in scope). + `setup-node` v4→v6. Two commits; schedule re-registers on merge to `development` (first run next Sunday).
- **Colobbo-Mobile → [PR #366](https://github.com/Colobbo/Colobbo-Mobile/pull/366)** (development / @pjtech18). Scan GREEN; only the preventive `setup-node` v4→v6 (checkout already `@v6`). No lockfile change. Edited via GitHub contents API (local clone fetch hung); committed content verified as exactly the 1-line change.
