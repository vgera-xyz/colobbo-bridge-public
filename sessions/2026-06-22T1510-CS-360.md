Shipped **M7** of the CS-373 Phase-1 sequence — which the derivation result §7 resolved (per Q-P4-1) to be **CS-360 the gate, PULLED FORWARD** ahead of the M8 consumer-side pin migration ("gate-before-migrate": the gate must be in place + complete before mass migration runs against it). M7 enhances the existing cross-Lobe contract gate `scripts/verify-lobe-compat.sh` (Gate 2) — a **gate extension, not a new gate**, exactly as the CS-360 ticket prescribes. PATCH ship (gate-only; no Lobe / schema / glaze-behaviour change), matching the CS-385/M3 gate-addition-as-PATCH precedent.

## What was done — CS-360 Assertion A (three enhancements to `verify-lobe-compat.sh`)

1. **Completed the REGISTRY (10 → 21 entries).** Pre-M7 the hardcoded REGISTRY covered only **10 of the 23** glaze `consumes_*_schema` pins. The **13 unverified** included **all 8 of Code Reviewer's pins** (the ticket's named "biggest single hand-maintenance point"), both Test Writer pins, both Doc Writer pins, and the 2 retired-migration sentinels. Registered the **11 live-producer pins** (review←{planner,arch,frontend,backend,database,mobile,security}, test←{arch,review}, docs←{review,arch}); all 11 verified currently-passing at registration.
2. **Completeness forcing-function.** Every glaze `consumes_*_schema` pin must now be **either registered OR in an explicit `EXCLUSIONS` list** (the 2 retired-migration sentinels — no live producer schema per ADR-037 / CS-368, preserved as sentinel-input contracts). A future glaze pin added without registration now **FAILS** the gate. This closes the silent registry-vs-glaze drift class that let the 11 pins go unverified (added incrementally CS-138→CS-145, never registered) — it is the literal "forcing function for a propagation obligation" CS-360 names as its thesis.
3. **Accumulate-all-failures (no short-circuit).** The gate previously `exit`ed on the first failing entry; it now collects **every** failure across all entries + the completeness pass and reports the full list in one run (the ticket's core "list ALL stale consumers in one tag-run" ask). Multi-failure exit-code precedence: **2 (missing file) > 1 (config/wiring/completeness) > 3 (version-drift)** — most-fundamental first.

## CS-360 Assertion B — NOT built (subsumed by Gate 14)

The post-glaze-edit compose-drift detector is **structurally subsumed by Gate 14** (`scripts/verify-compose-fixtures.sh`), which recomposes all `agents/global/*` and byte-diffs against the committed copies, and runs on **PRs touching `glaze/*.json`** (post-CS-194 paths trigger), **at every tag**, and in the **local pre-tag checklist** — catching exactly the ticket's Failure-2 scenario at all three points. Every `{{glaze.x}}` substitution flows through compose → Gate 14; there is no compose-drift case it misses. Building a separate detector would duplicate Gate 14, against CS-360's own "implement as gate extensions, not new gates" instruction + the CS-22 non-duplication discipline. Documented in the script header + CHANGELOG so it reads as a deliberate decision, not an omission.

Confirmed necessary.

## Lesson captured

The ABSORB step (per the CS-360 ticket's own methodology) surfaced that the gate covered only 10/23 pins — a fact invisible from the ticket text alone (the ticket cited "14 entries × 2 glazes," a count that predates the ADR-037 migration/regression retirement). Reading the actual REGISTRY array vs the actual glaze pins is what turned M7 from "make the gate accumulate" into "the gate was 57% blind; complete it AND add the forcing function so it can't go blind again." Completing the coverage was the larger, higher-value half — and exactly the drift-class CS-360 exists to kill.
