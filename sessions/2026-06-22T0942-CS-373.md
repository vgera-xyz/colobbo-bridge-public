# CS-373 M5 (op #38) ‚Äî read-side entity-grounding gate shipped (tag v2.47.0)

## Summary

Shipped **M5** of the CS-373 Phase-1 sequence ‚Äî op #38 ("ground a claim against the live repo"), the semantic-gate ground-claim extension. New `scripts/ground-claims.mjs` mechanises the operator-grep that previously happened by hand (the `tblInspection.status`-class check before approving a Backend GROUP BY): at the cibo-pipeline **S3 semantic gate** it grounds read-side persistence_op `table_or_sp` claims (`op ‚àà {select, sp_call}`) against the live schema source, emitting `{grounded, provably_absent, unverifiable, advisory_writeside}`. Control-flow is **verifiability-outer-gate**: `provably_absent` (readable source) ‚áí feature-pipeline ¬ß6 bucket-3 bounded-loop=1 then halt (never auto-substitute); write-side ‚áí advisory (greenfield-emit per CS-216 means Database may create the entity downstream, AFTER Backend in the DAG); `unverifiable` (stale/absent/unresolvable source) ‚áí advisory ‚Äî a high-trust source-class can NOT escalate an unverifiable into a halt.

The build also delivered a **step-0 prerequisite + latent-bug fix**: `glaze.repos[*].path` was hardcoded to `~/Desktop/Colobbo/...`, stale since CS-320 moved the live tree to `~/dev` (it silently resolved to the 16-day-old iCloud backup). Repointed to `${COLOBBO_ROOT}`/`${FINTECH_ROOT}` env-root indirection, taught six consumers to expand `${VAR}` Ware-agnostically, and added `verify-glaze-shape.sh` invariant **(ad)** to prevent re-drift. This **also fixed M2's `installed-package-probe.mjs`**, which had been grounding against the stale backup.

Scope was deliberately **narrow** (operator decision, reversed from "wide" after the architecture read): read-side `table_or_sp` only. The CS-322 source-enum strictness item is **unfalsifiable under the current stale graphify graph** (every code-symbol claim ‚Üí `unverifiable`), so it was NOT built ‚Äî noted on CS-322, gated on CS-388.

## Lesson captured

A `${VAR}` indirection token that contains a product name (`${COLOBBO_ROOT}` ‚Üí "colobbo") trips the Gate-5 / three-layer forbidden-token grep. The **consumer** must therefore expand `${VAR}` generically (read the var name, resolve from env) and the product token must live **only** in the glaze (the Ware layer). This is why the expansion code never names `COLOBBO_ROOT`.
