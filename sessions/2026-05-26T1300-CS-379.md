## Substrate findings worth memory updates

1. **GitHub Actions service incident at workflow-dispatch granularity** (2026-05-26, ~10:54–12:49 UTC ish): push events received, refs correctly updated, permissions enabled — but workflow evaluator silently dropped triggers. workflow_dispatch API itself returned HTTP 500. Status page lagged reality (operator hypothesis confirmed empirically by retry success). Pattern recovery: poll-the-status-page is unreliable; retry-the-dispatch is the empirical test. Worth a memory: `feedback_gh_actions_dispatch_real_test_not_status_page` — when GitHub Actions appears down, check via direct dispatch (not status page); status page lags by minutes; operator caught this in 30s by retrying.

2. **Schema-validator skeleton-wrap `advisories: []` requirement** — when wrapping a partial output (just the new field) in a full schema's required-field-skeleton for validation, missing required scalars / arrays are caught at validation time. Worked example: my first skeleton attempt failed on `advisories` (required field omitted); adding `advisories: []` fixed it. The skeleton's purpose is "minimum valid ReviewResult that wraps the field under test" — knowing the FULL required-field set requires reading the schema, not just the field-under-test. Pattern for future schema-validator skeleton-wraps: enumerate all required fields from the schema once + cargo the skeleton; iterate on errors.

3. **Projection-not-byte-compare as the load-bearing design pattern for deterministic gating of LLM-or-script structured output** — operator's pushback on the original tautological-byte-compare-with-resnap design surfaced this. The principle: gate the BOUNDED fields (enums, IDs, sets) that are hand-derivable from the algorithm + fixture; leave free-form prose ungated for byte-compare BUT schema-validated for shape. This avoids "the gate gates its own snapshot" anti-pattern. Worth a memory: `feedback_projection_not_byte_compare_for_structured_output_gates` — when gating structured output with mixed-bounded-and-prose fields, project bounded fields for byte-compare + schema-validate the full output separately; never re-snap prose-fields-of-PINNED-goldens.

4. **Design-lock stale-text grep applies to plan-mode revisions too** — operator caught me missing this after a multi-pass plan revision left Step 6 soft-sanity + Verification #8 + discharge table stale on the dropped re-snap / message-template design. This reinforces the existing `feedback_design_lock_stale_text_grep` memory: applies to (a) /rthink Pass 2, (b) plan-mode mid-pass revisions, AND (c) Linear ticket descriptions when ticket is in scope of the design lock. Worth updating the memory to broaden applicability.

5. **CR LOBE.md "Version history" section is SCHEMA-version-only** (1.0.0/1.1.0/1.2.0) — surprised me at Step 3; M1 (CS-378) also didn't add a Lobe-version-history section + didn't update the prose Status header at v0.3.0. The Lobe version is captured authoritatively in `lobes/_registry.json` + the frontmatter — no separate prose log.
