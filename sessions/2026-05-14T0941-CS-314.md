# CS-314 — AI review hardening ship

## Summary

Shipped CS-314 hardening of the AI review pipeline + bundled CS-310 single-quote-pattern hygiene end-to-end. Replaces the PR-comment-canonical AI review surface — defeated in practice by developers deleting Claude's bot comments week-of-2026-05-12 — with **GitHub Check Run as the new canonical, deletion-resistant surface** (precedent: CodeQL, SonarCloud, Snyk). Adds two more surfaces: `workflow_dispatch` manual trigger for re-reviews without close/reopen hack, and Slack mirror to `#cibo-asks` with dynamic `<@U…>` mention of the currently-assigned PR reviewer at runtime via the new `.github/workflows/slack-user-map.json`. PR comment preserved as third additive mirror — powers CS-290 Feature 3 (re-review historical-context via `<!-- ensemble-review-v1 -->` marker).

End-to-end validated across **7 successful `workflow_dispatch` smoke runs** on web-angular smoke PR #514 (now closed). 9 of 9 acceptance criteria hard-verified including AC #3 comment-deletion drill (Check Run survives `gh api comment DELETE`) and AC #6 multi-reviewer + no-reviewer-fallback edge cases. Two false-positive 3-model agreements surfaced during smoke (ensemble claimed `actions/checkout@v6` invalid and `__typename` field absent — both demonstrably false) — expected meta-output demonstrating exactly the human-in-loop value the `verdict=attention` signal is designed to provide.

Tagged **v2.20.0**. Three product-repo PRs raised, **awaiting reviewer merge per CS-290 precedent**; CS-314 stays In Progress until all three land. CS-310 closed as Done per the user's explicit pattern-hygiene narrative.

## Smoke-phase surfacings — 4 real defects + 2 patterns codified

Each surfaced during web-angular smoke validation, each fixed-forward before product rollout:

1. **Slack payload missing newline** (run 2): bash `$(printf '\n')` strips trailing newline → `<URL>PR:` concatenated. Fix: brace-group printfs (`4645170`).
2. **Annotation parser too strict** (run 3): re-review models emit `~line 349-383`; original regex rejected all. Fix: tolerant parser stripping common decorations (`04f85e5`).
3. **Pattern T.3 — display-name divergence** (Slack ID pre-flight): map entry for pjtech18 resolved to "Colobbo Support" surface name → looked wrong but legitimate (Jaydeep uses support@colobbo.com). Verification discipline: trace through git log committer email / CLAUDE.md role mapping, never surface display name. Codified in state-drift-patterns.md.
4. **Pattern T.4 — webhook destination integrity** (smoke runs 1-5): Slack incoming webhooks are channel-bound at install; payload cannot override. Org `SLACK_WEBHOOK_URL` bound to `#colobbo-web`, not `#colobbo-radar` as plan assumed. 5 smoke posts landed in wrong channel before user-side observation surfaced the gap. Fix: dedicated `SLACK_CIBO_ASKS_WEBHOOK_URL` secret bound to `#cibo-asks` (the CS-260/CS-272 canonical Cibo→devs channel). Pre-flight rule: locate webhook secret + slack_search for prior posts (channel field reveals binding) before assuming destination.

## Two false-positive 3-model agreements — preserved as designed signal

The smoke PR's own workflow change triggered ai-review on itself across 7 runs. Two findings achieved 3-model agreement (high-signal classification):

- "`actions/checkout@v6` is not a valid version" — but v6 ran successfully end-to-end in same workflow (used by setup + aggregate jobs); demonstrably false.
- "`select(.__typename == \"User\")` will silently return empty array because gh pr view (REST API) doesn't include __typename" — but Pattern T.1 pre-flight probe confirmed gh pr view --json reviewRequests DOES return `__typename` (GraphQL-shape, not REST); smoke runs 1-7 all resolved reviewers correctly.

Both false positives are good data. Verdict `attention` is the correct signal — the human-in-loop sees the disagreement-via-confident-error and the workflow's design goal of "high-signal, not low-noise" is satisfied. This argues against CS-306's severity-threshold blocking landing prematurely without sufficient ensemble signal-quality data — premature blocking on 3-model agreement would have failed this very PR.

## Post-PR-raise surfacing — Sentry catches what /rthink missed (Pattern T.5)

After the initial CS-314 ship landed and the 3 product PRs were raised, Sentry reviewed PR #515 and flagged `MED #515` on `.github/workflows/ai-review.yml` line 141. The bug: `git diff "$PRIOR_SHA" "$PR_SHA"` works under `pull_request` event (PR head + ancestors are pre-fetched by default checkout) but silently degrades under `workflow_dispatch` event (default-branch-tip only). Pre-existing `git cat-file -e` guard would skip the diff with no error → Feature 3 historical-context re-review degrades to empty, completely silently. Smoke test would have failed to surface this because all smoke runs were on the same SHA (the prior comments existed at the same `<!-- reviewed-sha: ... -->` marker each time, and the diff was effectively empty regardless).

Fix landed as 3-line surgical addition (2 explicit `git fetch origin "$SHA"` + 1 comment) before the existing guard. Chose `git fetch` over Sentry's suggested `gh pr checkout` to keep the change surgical — no working-tree mutation, no side effects on downstream steps in the same job. Existing `git cat-file -e` guard preserved as defense-in-depth.

Propagated to all 3 product PRs as PR-branch commits (auto-update):
- web-angular #515 → `d858ee87a`
- api_node #243 → `996219b`
- Colobbo-Mobile #351 → `d2afb9e9`

Reply posted to Sentry's review comment on PR #515 (https://github.com/Colobbo/web-angular/pull/515#discussion_r3240535996) — thanked the reviewer, explained the surgical-fetch choice over working-tree-mutation alternatives, referenced the Pattern T.5 codification.

**Cross-system layered-defense — this is the CS-290 lesson, again.** /rthink's 10-point pass cleared the canonical change (checked Pattern T.1 + T.2 + T.4 derivatives, REVIEWERS_JSON env-block hygiene, payload-construction discipline, repo-aware CLI flags) but never asked "does this CLI invocation behave the same on every event type that reaches this code path?". Sentry independently flagged it because Sentry's reviewer didn't share /rthink's framing — its model came at the workflow from a different angle and saw what /rthink missed by construction. The strategic-level fix is investing in diverse validation surfaces, not perfecting any single discipline (per CS-290 bridge § Layered defense). T.5 codification updates /rthink point (j) implicitly — future workflow audits should explicitly probe event-context state under EVERY trigger type.
