# CS-371 — Fixture metadata convention + v2.44.1 ship

## Summary

Fleet-wide substrate ship that codifies the underscore-prefixed fixture-author-metadata convention at the schema layer. Discovered when this session's earlier validate-doc-writer.mjs smoke surfaced the original DocResult fixture's top-level `_description` field failing `additionalProperties: false`. Verification revealed the surface was vastly larger than session-end estimate: **88 fixtures across all 11 Lobes carry `_description`**, plus **10 distinct underscore-prefixed metadata keys** (`_canonical`, `_changelog_emitter_deferred_to`, `_description`, `_emitter_deferred_to`, `_glaze_assumptions`, `_lobe_version_target`, `_methodology`, `_note`, `_product`, `_version`) are in active use. Three fix options weighed; **Option B (schema-level `patternProperties: { "^_": {} }`)** selected per Raku-extractability lens (fixture authors in any Ware inherit the convention via the schema template; preserves docs; future-proofs the 10 metadata keys; preserves the typo-catching `additionalProperties: false` contract for non-underscore extras).

Operator delegated two judgment-calls explicitly ("recommend what YOU assess is right, push back if I'm wrong, don't just comply"): Q1 — memory-lobe documentation surface (CC chose schema `$comment` over separate wiki doc, refining operator's lean); Q2 — tag mechanics (CC verified Pattern-S + registry-regen prerequisite, agreed with operator's lean). Both decisions executed with explicit-agreement-with-refinement, not silent rubber-stamp.

Single substrate commit: `e72348348` — 25 files changed, +364 / -41. Substrate + VERSION + registry + wiki doc + glaze canonical_pages. Tag v2.44.1 to follow this bridge entry for 18-gate Lobe Regression CI ratification.

## What was done — six workstreams

Exhaustive scan of underscore-prefix top-level keys across fixtures returned 10 distinct conventions: `_canonical`, `_changelog_emitter_deferred_to`, `_description`, `_emitter_deferred_to`, `_glaze_assumptions`, `_lobe_version_target`, `_methodology`, `_note`, `_product`, `_version`. No Lobe output-schema currently declared any underscore-prefix property; all 10 schemas used `additionalProperties: false` uniformly (memory-lobe omits the key entirely — JSON-Schema default-permissive). No wiki doc currently codified the convention.

### 2. Three options weighed; Option B selected

- **A** — strip `_description` from 88 fixtures + the other 9 `_*` keys would still break. Reactive. Loses 88 author-docs. (Operator-framed; CC concurred this is the wrong shape.)
- **B** — schema-level `patternProperties: { "^_": {} }` accommodation. Universal; future-proof; Raku-extractable; preserves contract; ~21-file ship. **Selected.**
- **C** — strip from Result-affected fixtures only (~7 files). Partial; splits convention across fixture types; still reactive. **Rejected** (inconsistency cost > scope-creep cost).

Operator-approved Option B with two judgment-call delegations:
- Q1 (memory-lobe documentation): operator leaned "document the inconsistency, don't leave it silent." CC refined: document via schema's `$comment` field (in-place propagation), not a separate wiki doc.
- Q2 (tag v2.44.1): operator leaned "tag for 18-gate ratification." CC verified Pattern-S mechanics + the `feedback_version_bump_requires_registry_regen` prerequisite, agreed.

### 3. Schema substrate (11 files)

- `lobes/mobile-lobe/output-schema.json`
- `lobes/planner-lobe/output-schema.json`
- `lobes/security-lobe/output-schema.json`

Single Edit pattern applied 10x for standard Lobes (identical old/new strings, different file paths). Memory-lobe got 2 edits (existing `$comment` extension + `patternProperties` block addition with different anchor since no top-level `additionalProperties`).

### 4. Validator substrate (10 files)

Extended each `scripts/validate-<lobe>.mjs` minimal-JSON-Schema validator to support `patternProperties`. Three validator shapes existed pre-fix:

- **Shape 1** (8 validators: arch/backend/code-reviewer/database/doc-writer/frontend/plan/test) — identical leading `// additionalProperties (objects)` comment + two if-blocks. Identical extension applied via shared anchor.
- **Shape 2** (validate-mobile.mjs) — same code, no leading comment. Same extension; slightly different anchor.
- **Shape 3** (validate-security.mjs) — compact nested structure inside `properties` block, `if/else if` style + variable name `k` instead of `key`. Restructured inside the existing nesting to add patternProperties handling.

The new code tracks pattern-matched keys in a local `Set`; matched keys are excluded from the `additionalProperties: false` check (per JSON Schema Draft-07 spec). Backward-compat preserved: validators without `patternProperties` in their schema behave identically.
