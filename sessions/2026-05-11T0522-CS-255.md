# CS-255 ship — graphify continuous refresh hardening

## Session context

Three product-repo graphs (`graphs/api_node`, `graphs/web-angular`, `graphs/Colobbo-Mobile`) were stale (mtimes Apr 19–May 1) because the `graphify-refresh.yml` and `graphify-refresh-self.yml` workflows fail when two graphify runs try to push to `colobbo-agent-system` `main` concurrently. Confirmed root cause: non-fast-forward push rejections (`! [rejected] main -> main (fetch first)`) — three of four open auto-issues (#14, #13, #12) showed the exact `(fetch first)` error in their workflow logs; issue #2 logs were retention-expired but symptom-consistent.

This ship goes beyond the immediate push-race fix to address all four CS-255 lessons together: (1) push-race recovery via retry-with-rebase, (2) wasted-run reduction via receiver-side path-filter early-exit, (3) weekly cron baseline, (4) drift detection via a new monitor workflow. All four lessons shipped in this session as atomic commits.

## What shipped — CS-255

### 1. Retry-with-rebase loop (commit `c094528`)

Edits to BOTH `.github/workflows/graphify-refresh.yml` and `.github/workflows/graphify-refresh-self.yml`. The "Commit graph changes" step now wraps the push in a 5-attempt loop with 1–5s jitter that pulls-with-rebase before each retry. Recovers from cross-workflow push contention when product-repo dispatches and cibo self-refreshes land within seconds.

Concurrency-group widening (alternative to retry) was **considered and rejected** — would serialize cibo self-refreshes behind product-repo refreshes, slowing feedback on agent-system pushes. Rationale documented inline above each `concurrency:` block in both workflow files; CHANGELOG `Fixed` entry; this section. Trigger surface for reconsideration: drift-monitor firing repeatedly despite retry-with-rebase. NO follow-up ticket — rationale is durable and discoverable.

### 2. Path-filter early-exit step (commit `79740cf`)

New step "Path-filter early-exit (dispatch only)" in `graphify-refresh.yml`, gated on `if: github.event_name == 'repository_dispatch'`. Runs `git diff --name-only TARGET_SHA~1 TARGET_SHA` against the workflow's `scope` output. If zero overlap, sets `SKIP_GRAPHIFY=true` env-flag; downstream steps (Setup Python, Install graphifyy, Run AST refresh, Commit) gate on `if: env.SKIP_GRAPHIFY != 'true'`. Job completes green with no Python install + no commit.

target-repo checkout `fetch-depth` bumped 1 → 2 to make `~1` resolvable. First-commit / shallow-fetch / force-push edge cases fall through to running graphify (no false skip). `workflow_dispatch` and `schedule` events bypass the filter (always run — explicit/baseline signals).

### 3. Weekly cron baseline (commit `fa2459a`)

New `weekly-trigger` job in `graphify-refresh.yml` (gated on `if: github.event_name == 'schedule'`) that fires 3 `workflow_dispatch` events for `api_node`, `web-angular`, `Colobbo-Mobile` via `actions/github-script`. Reuses existing `refresh` job unchanged per dispatch — no matrix-strategy complexity. Cron: `0 14 * * 0` = Sunday 14:00 UTC = Monday 00:00 AEST / 01:00 AEDT (start of Melbourne work week). `actions: write` permission set at job level.

Colobbo-Web excluded — .NET decommissioning, not a Phase 3 dependency surface.

### 4. Drift-monitor workflow (commit `3b49e46`)

New file `.github/workflows/graphify-drift-monitor.yml`. Daily 22:00 UTC schedule (= 08:00 AEST / 09:00 AEDT next day, start of Melbourne workday). For each in-scope product graph (`graphs/<repo>/graphify-out/graph.json`), derives last-refresh time from `git log -1 --format=%ct` (NOT mtime — `actions/checkout` sets mtime to checkout time, so mtime-based detection would always read AGE ≈ 0). Opens deduped issues if >threshold days old.

Dedup via title-exact server search + `jq` client-side exact-match. Title format: `"Graphify graph stale for <repo> (>N days)"`. Labels: `graphify` + `graphify-drift`. Issue body includes likely causes + manual-recovery `gh workflow run` command.

`workflow_dispatch.inputs.threshold_days` (default 7) supports smoke testing via `-f threshold_days=0` force-alarm. Implementation uses bash `${VAR:-7}` fallback (NOT `${{ inputs.threshold_days || 7 }}` — GitHub Actions expressions treat `0` as falsy, would override smoke value back to 7). Checkout uses `fetch-depth: 0` for `git log` against the graph file.

Out-of-band: `graphify-drift` label created via `gh label create` before drift-monitor first runs (otherwise `gh issue create --label graphify-drift` would fail).

## Schedule rationales

- **`0 14 * * 0` weekly-trigger** = Sunday 14:00 UTC = Monday 00:00 AEST / 01:00 AEDT. Start of Melbourne work week — refreshed graphs available before engineers start work Monday.
- **`0 22 * * *` drift-monitor** = daily 22:00 UTC = 08:00 AEST / 09:00 AEDT next day. Start of Melbourne workday — drift surfaces in time for engineer triage.
- **Cross-check vs existing self-refresh `0 3 * * 0`** (Sunday 03:00 UTC = Sunday 13:00 AEST): 11h gap to weekly-trigger minimizes cron-driven contention window.

## Implementation caveats discovered + worked around

1. **`actions/checkout` sets file mtime to checkout time, not commit time.** mtime-based drift detection would always read AGE ≈ 0. Implementation pivoted to `git log -1 --format=%ct -- graphs/<repo>/graphify-out/graph.json` for true last-refresh time. Required `fetch-depth: 0` on the checkout step.
2. **GitHub Actions expressions treat `0` as falsy.** `${{ inputs.threshold_days || 7 }}` would override the smoke-test value `0` back to `7` — defeating the smoke. Implementation uses bash `${VAR:-N}` fallback (only triggers on unset/empty, NOT on `'0'` which is non-empty in bash).

## Push-race in-the-wild evidence (eat own dogfood)

During this session's `git push origin main` (after locally committing all 4 commits), the push was REJECTED with `! [rejected] main -> main (fetch first)` — exact same failure mode CS-255 fixes. Recovered via `git pull --rebase origin main && git push origin main` (the manual version of what the new retry loop does). Two product-repo refreshes (`7ccf7e2`, `1c52cc1`) had landed on remote during my local commit time. Rebase replayed cleanly (different subtrees: `.github/workflows/` vs `graphs/<repo>/`). Live demonstration of (a) the failure mode being real, (b) the rebase-then-push recovery being correct.
