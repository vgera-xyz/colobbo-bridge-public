# COL-79 smoke → substrate version-self-report evolution

## Summary

One coherent arc. The COL-79 Planner functional smoke (prior session,
`wiki/bridge/sessions/2026-05-19T0550-COL-79.md`) surfaced two latent
substrate defects: (1) Glaze **key vs name** identifier ambiguity, and (2)
Planner **runtime version drift** (deployed agent self-reported `1.0.1`
against a LOBE.md frontmatter saying `1.0.2`). This session resolved both
and generalised the fix fleet-wide, in three pieces plus an opening
4-item warm-context batch. COL-79 the *feature* remains **parked** (not
built/released — revisit after CS-346); only its smoke-surfaced substrate
findings were resolved. Tomorrow CS-345 starts fresh.

## The arc (chronological)

**4-item substrate batch** (`cfe3bf4b2`) — Glaze key-vs-name conventions
documented in `wiki/architecture/lobe-substrate-layers.md`; post-compact
hook line cap 100→200 (`.claude/settings.json`, CS-208 alignment);
Context7 version-pinning rule (`.claude/rules/context7-usage.md`, CS-347
Stage 1); Planner version drift **identified** (root-cause investigation,
no fix in batch). Context7 project-scope enablement landed earlier same
day (`3e4eb68d4`, `9e01e4e34`, CS-347).

**Piece A** (`702d66292`) — FORWARD fix for the version drift. Root cause:
the `1.0.1→1.0.2` bump shipped in CS-138 Phase-2a (`74ad21ddb`) as a real
Layer-0 emit-discipline migration but with **no `## 1.0.2` CHANGELOG
entry**; the deployed agent self-reports from changelog narrative (compose
stripped frontmatter `version:`), so it emitted the last documented
`1.0.1`. Added the 1.0.2 CHANGELOG entry (dated 2026-05-18, when it
actually shipped) + recomposed `agents/global/planner.md`. Loop closed:
runtime now self-reports 1.0.2.

**Piece B** (`fcaed3eac`) — the architectural generalisation (Deep-A).
`scripts/compose-lobes.mjs` now injects `version:` (from LOBE.md
frontmatter via `fmVersion()` — the same source the registry
`composite_signature` hashes) **and** `schema_version:` (from
`output-schema.json` `_metadata.current_version`) into **every** deployed
agent's frontmatter; methodology prose instructs the runtime to emit both
from its own frontmatter, never inferred. Fail-loud (new exit codes
14/15) — no silent `0.0.0`. Fleet-wide: 12 Lobes, 27 files. Symmetric
`schema_version` twin defect (R4) fixed in the same change. memory-lobe
has no payload version field (defect N/A) but its wiki-page `lobe_version`
*stamp* was anchored to its own frontmatter — full coverage, no shortcut.
Durability: the cross-cutting prose lives once in Layer 0
(`lobes/_shared/lobe-frontmatter-base/1.0.0.md`, zero pinners → safe
in-place edit, no rollover); `scripts/new-lobe.sh` already scaffolds the
correct sources so every future Lobe inherits the contract by
construction. `deploy.sh` run — **129 deployed**, live agents verified
(`~/.claude/agents/planner.md` → `version: "1.0.2"`).

**Piece C** (`ec3338aa6`) — premise inverted. Piece C was scoped to
"reconcile fixture versions to current source of truth." Investigation
found the fixture `*_version`/`schema_version` values are **not stale
drift — they are deliberately frozen regression anchors**, triple-encoded
(`cases.json` jq `== "X"` asserts + `"hardcoded at X"` desc text +
`*_version_tested` header). Runtime tracks HEAD by design; goldens stay
pinned by design — they *should* diverge. Reconciling would be 100+
lockstep edits across 4 surfaces in 11 Lobes, destroy the anchors, create
a per-bump treadmill, for zero correctness gain. Action (Option α): one
doc subsection in `lobe-substrate-layers.md` stating the by-design pin +
a Phase-C forward hook. No fixture touched.
