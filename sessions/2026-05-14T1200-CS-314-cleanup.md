## Tag immutability via merge, not rebase + force-tag

When the main push was rejected at Phase 4 step 2 due to origin/main advancing with 2 graphify-cron commits, the decision was between:

- **Option 1: Rebase + force-update tag** — `git rebase origin/main` → `45f6d77` gets a new SHA → `git tag -f v2.20.1 <newSHA>` + `git push --force origin v2.20.1`. Linear history. **Destructive** action on a published tag artifact.
- **Option 2: Merge origin/main** — `git merge origin/main` → creates merge commit. Tag stays at original `45f6d77`. Non-destructive. Linear history slightly noisier (merge commit for graphify-cron interleave).
- **Option 3: Wait for CI, then merge** — chosen.

Chosen Option 3 because the in-progress CI run was about to deliver validation evidence at `45f6d77`. Force-updating the tag would throw away that validation. Merge preserves the tag at the validated SHA at the cost of one merge commit in main's history. Pattern G overlap check (`comm -12` between my-commit's filelist and origin-commits' filelist) confirmed zero file overlap → clean merge guaranteed before invocation.

Recorded as a worked example for **future divergence resolution**: when a tagged commit cannot fast-forward, prefer merge over rebase+force-tag if CI is already validating the original SHA. The merge cost is one commit; the rebase cost is throwing away validation evidence.
