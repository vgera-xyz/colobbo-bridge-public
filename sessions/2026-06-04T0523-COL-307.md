## What was done

### Task 3 — WP summary-grid (`76f7630`)
1. **Kernel refactor — `LINK_GRAPH` single-source.** Added a `LINK_GRAPH` constant (R1_ACTION / R1_INSPECTION / R2_ACTION module pairs) consumed by BOTH the flat path and the new grouped path; the flat path's `resolveLinkedEntityIds` + its 28 tests are **untouched** (definition-level single-source — full edge-resolver unification was rejected because it would break the Phase-1 `.select`-count assertions).
2. **New grouped kernel fn `aggregateGroupedByWorkPackage`.** ONE `mssql.execute` of a `GROUP BY workPackageId` CTE (built from a shared `bidirectionalEdge` SQL fragment): per-WP `actionCount` + **action byState (A/I/C → toDo/inProgress/complete)** + `inspectionCount`, `R1∪R2` deduped by record identity (`UNION`/`DISTINCT (wpId, entityId)`), `ORDER BY <allowlisted count> OFFSET/FETCH` at the DB. WP `name` joined in (display + DB-side name-sort). **Claim OMITTED** — tblClaim is PostgreSQL (see 3b).
3. **Candidate seam — reuse, don't reinvent.** `workPackageGridService` resolves the project-scoped, tenant-safe, EAV-filtered candidate WP id-set via the EXISTING filter authority `reportingService.runWorkPackageView` (→ `SP_WorkPackageView_Run`), forwarding `filters_json` verbatim and applying row-level scope through the **SHARED** `mergeScopeIntoFilters` (`permissions/scopeToFilters`) — zero EAV/scope logic forked into the grid. An under-fetch guard throws rather than truncating the all-ids pull.
4. **Endpoint** — `workPackageSummaryGridController` (Zod, identity/scope from JWT+recordScopeMiddleware only, sort allowlist, base64 `filters`), `workPackageSummaryGridRoutes`, mounted in `index.ts`. `{ success, data:{ items, pageIndex, pageSize, total, totalPages } }`.
5. **Tests** — 4 grouped-fn unit tests + 7-case supertest e2e (envelope, projectId-required, IDOR company/scope passthrough, pagination metadata, sort-allowlist reject).

### Task 3b — flat-kernel claim cross-DB fix (`9580edb`, separate commit)
- **The bug:** shipped `aggregateOverAnchorSet` queried claims against MSSQL (`getMssqlDb()` + `tblClaim`), but **claims live in PostgreSQL** (COL-206; confirmed by `mssql-schema.ts` has no tblClaim, `modules/claim/claim.schema.ts` is PG, `status.service.ts:247` "broken SQL Server lookup for Claim", `connectToPg()`). The MSSQL query hit a non-existent table → **prod per-WO/dashboard claim counts wrong (likely 0)**, masked by 28 mocked-MSSQL tests.
- **The fix:** claims now queried from PostgreSQL via the same `db` the kernel already uses for `activityLog` (the dual-DB stitch) — `claim` by `workOrderId`, defense-in-depth `companyId` filter; one PG query yields ids → count = ids.length + feeds the activityLog 'Claim' entity filter. MSSQL claim selects removed.
- **Test surgery:** claim seeding moved MSSQL→PG queue across all 3 flat-path suites; MSSQL select-count assertions dropped 2 (10→8, 6→4); a **NEW test** ("counts claims from PostgreSQL, NOT MSSQL") seeds claims only in the PG queue and **fails if claims ever revert to MSSQL**.
