## Root-cause hypotheses (unranked — full table in the report)

- **H1 iCloud conflict-copy** — every observed signature present (byte-identical, 0600 perms, hits node_modules/build/never-CC-written trees, whole directories, active `brctl` container). Strongest fingerprint; **not operator-confirmed**.
- **H3 CC tool-retry / H6 LLM-hallucinated filename / H5 git op** — disconfirmed by mechanism (paths CC never writes; byte-identical; no git codepath produces `name 2.ext`).
- **H2 Finder Keep-Both / H4 parallel CC writes** — cannot explain bulk/dir-level; open whether H4 *aggravates* H1 (a CC write mid-iCloud-sync-up is exactly H1's conflict trigger).
- **H7 Time Machine / APFS local snapshots** — CS-320 asks to investigate; **NOT yet correlated** (`tmutil listlocalsnapshots` + `bird` timing = a follow-up discovery pass if operator wants it).

## Hard structural constraint surfaced (hook-design space)

The dup-creation event is **out-of-band**: `bird`/iCloud (or APFS) creates the file, **not any Write/Edit/Bash tool call**. **No PreToolUse/PostToolUse tool-matcher hook can intercept it** because CC never issues the creating tool call. This bounds any future structural fix to: non-tool-coupled detection events (`SessionStart`/`Stop`/`SubagentStop`), a check on an operation CC *does* run (pre-`git add`/commit), or environmental remediation (repo location vs iCloud; the partial `.gitignore` mask). D3 precedent check: NO existing hook (of 13) blocks file-creation by pattern; no sweep/cleanup script exists in `scripts/`. Prior responses were all manual (COL-280/284/285 `find -delete`, RKU-78 gitignore-PR).
