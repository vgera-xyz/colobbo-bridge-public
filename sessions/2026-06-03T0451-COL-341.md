# COL-341 vs COL-307 — connected-record rollup: read-side aggregation over row materialisation (investigation only)

## Summary

Read-only architecture review (an `/rthink`-shaped investigation, xhigh effort) of **COL-341** (In Progress, authored by Jignesh) before any of its code is committed. COL-341 proposes materialising WorkOrder-hierarchy rows into `tblLinkModule` on every link create/delete (walk `parentWorkOrderId` to root, insert the `sourceSet × targetSet` cross-product), to let a Work Package grid filter/sort "outstanding actions across all child WOs." We evaluated a normalised alternative (closure table + count-by-type read-side aggregation) and **recommend Path B (read-side aggregation), starting B1 (no new table). Confidence HIGH. Reject COL-341's materialise design.**

**No ship this session** — investigation + recommendation only. Zero code, zero migration, zero ADR file, zero Linear edits. Deliverable is a pre-call artifact for the operator's upcoming Jignesh discussion. The full plan + decision brief + a staged-but-unposted COL-341 draft comment live in the local plan file `~/.claude/plans/shimmering-floating-hearth.md` (not in git — substance mirrored here for durability).

## What was investigated (VERIFY — TIER-1, quoted from disk; verified directly, not via subagent trust)

Three Explore agents fanned out across `api_node` (on branch `col-307-work-order-context-enrichment-backend`) and `Colobbo-Web`; every load-bearing claim was then re-read from the bytes by hand (standing distrust of free-form subagent filesystem claims).

1. **Re-parenting is a live write path in BOTH repos** — this is the load-bearing flag the brief asked to falsify, and it IS falsified:
   - api_node — `apps/api-node/src/modules/tracker/tracker.work-order.service.ts:273-300`: `updateWorkOrder()`'s mutable `intFields` literally contains `'workPackageId'` and `'parentWorkOrderId'` → `db.update(tblWorkOrder).set(patchValues).where(eq(tblWorkOrder.id, id) …)`.
   - legacy — `db/procedures/SP_WorkOrderInsertUpdate.sql:283` (`workPackageId = @workPackageId`) and `:310` (`parentWorkOrderId =@parentWorkOrderId`), in the UPDATE branch (the dominant WO-edit path today).
   - **COL-341's "Open Question (Resolved): No re-parenting scenario" is contradicted by source.**

2. **COL-307 (PR #250, Dev Done) already built the normalised aggregation machinery this grid needs** — `apps/api-node/src/services/workOrderSummaryService.ts`: `resolveDescendantWorkOrderIds(rootId)` (`:170`, downward `parentWorkOrderId` CTE), `resolveLinkedEntityIds(workOrderIds[], 'Action'|'Inspection')` (`:209`, bidirectional `tblLinkModule` UNION), and the shared set-shaped kernel `aggregateOverWoSet(workOrderIds: number[], authCompanyId)` (`:262`). These three are internal `async function` — **NOT exported** (only `getSummaryForWorkOrder` `:435` and `getSummaryForScope` `:456` are). Cheapest pre-merge tweak = export/extract the kernel.

3. **A normalised package rollup of the same shape already exists and works** — `apps/api-node/src/services/linkModuleService.ts:106-147`, `getWorkPackageLinkedActionIds(workPackageId)`: `WHERE workPackageId=@p0` + bidirectional `tblLinkModule` UNION → distinct action ids. Literally COL-341's grid goal, already solved without materialisation.

4. **No closure/path/hierarchy table exists** in either repo (api_node `db/mssql/schema.ts`; no `tblWorkOrder*Hierarchy/Closure` SSDT table) → a closure table would be new-build.

5. **No UNIQUE constraint on `tblLinkModule`** — Drizzle model `apps/api-node/src/db/mssql/schema.ts:213-219` has no `.unique()`, no DDL in repo. COL-341's retry-idempotency safeguard ("upsert/ignore on unique constraint") has nothing to enforce it.

6. **COL-341 is 0% committed** — `git log --all --grep="COL-341"` empty; no `*341*` branch; `resolveEntitySet` NOT FOUND anywhere. Nothing to unwind.

## Assessment + recommendation

| Dimension | Path A (materialise) | Path B (read-side) |
|---|---|---|
| Write amplification | HIGH (1 link → 4-6+ rows, hot path) | LOW (writes on rare tree change only) |
| Re-parenting safety | BROKEN (no WO-update maintenance; silent stale rows) | OK (computed on read, always correct) |
| Grid filter/sort | Doesn't help — still `GROUP BY`, over MORE rows | Feasible — reuse `listWorkOrders` pattern + aggregate cols |
| Shared-junction hygiene | Pollutes `tblLinkModule` (no `isHierarchy` flag; forces dedup on all readers) | Clean |
| Reuse of COL-307 | Rebuilds, more fragile | Reuses kernel |

**Operator ruling: "tolerate, don't assume"** — users likely don't re-parent routinely today, but the code permits it and it's a plausible cheap request; the system must not corrupt on re-parent. B is correct *regardless of frequency* because it computes hierarchy on read. Even absent re-parenting, B still wins on reuse + junction hygiene + the fact that pre-materialised rows give the grid no free sortable count columns.

**Path B1 (recommended next build, post-call):** export COL-307's kernel → new `GET /api/work-packages/summary-grid` endpoint, paginated + server-side filterable/sortable on counts + user-selectable module types, reusing the kernel and the `VALID_SORT_COLUMNS`/`buildWhereClause` pattern (`tracker.work-order.service.ts:39-82`). Closure table deferred to B2-only-if-profiling-demands. Separate endpoint from COL-307 (single-record detail ≠ paginated-grid), shared service kernel.
